Введение

Постепенное развитие процессоров привело к тому, что скорость выполнения инструкций стала на несколько порядков выше скорости доступа к оперативной памяти, в результате чего возникла необхоимость в увеличении кэш-памяти распологающейся непосредственно на самом кристалле, что в свою очередь привело к еe разделению на нексколько уровней. С учетом еще одного важного факта - прекращение роста тактовой частоты и увеличение количества ядер в процессоре, приходиться использовать раличные протоколы когерентности кэшей (MSI, MESI, MOESI, MESIF и др.), воспринимающие данные только блоками и не предусматривающие чтения и записи отдельных слов.

В добавок, для того чтобы дополнительно увеличить производительность, в процессорах активно используеются различные оптимизации исполнения, позволяющие выполнить наибольшее количество работы за один такт: предсказание ветвления, выполнение нескольких инструкций, внеочередное исполнение и пр. [1]

Кроме того перемещать инструкции может и компилятор, если решит что это является более эфективным и не меняет семантику программы. В этом случае, другие потоки могут увидеть запись, прежде чем она на самом деле должна произойти.

Разрешая компилятору, процессору либо среде исполнения выполнять операции в порядке, который является оптимальным с их точки зрения можно достичь более высокую производительность. Особенности исполнения и гарантии работы с памятью сосредоточены в трех фундаментальных свойствах: атомарность, видимость и упорядочивание, которые задают рамки возможных преобразований и определяют само понятие модель памяти [2].

В зависимости от гарантий и разрешенных преобразований в рамках модели памяти различают слабую (ARM-архитектура) и сильную (x86 - аритектура) модели [4].

Модель памяти Intel® 64 and IA-32

Прежде всего рассмотрим пример аппаратной модели памяти, которую предоставляет наиболее популярная архитектура x86.

Атомарность

Под атомарнстью понимается блок операций выполняемых как единое целое.

Префикс LOCK

Префикс выдачи сигнала LOCK# (префикс блокировки шины) заставляет процессор установить сигнал LOCK# во время выполнения следующей за ним инструкции. В многопроцессорной среде этот сигнал может быть использован для гарантии того, что процессор имеет исключительный доступ к любой памяти с распределенным доступом, пока установлен сигнал LOCK#.

Обратите внимание на то, что, в более поздних Intel 64 и IA-32 процессорах (в том числе Pentium 4, Intel Xeon, и процессоры семейства P6), блокировка может происходить без установленного LOCK# сигнала.

Префикс LOCK может предшествовать только со следующими инструкциями и только для тех их форм для которых операнд назначения является операндом в памяти: ADD, ADC, И, BTC, BTR, BTS, CMPXCHG, CMPXCH8B, CMPXCHG16B, декабрь, INC, ОТР, NOT, OR, SBB, SUB, XOR, XADD, и XCHG. Если LOCK префикс используется с любой из этих инструкций и операнд источника является операндом в памяти. может быть сгенерировано исключение (#UD). Это же исключение может быть с генерировано и в случае если LOCK префикс используется c инструкцией не из этого списка. Инструкция XCHG всегда производит блокировку шины, независимо от наличия префикса LOCK.

Инструкция XCHG

Команда XCHG осуществляет обмен значений двух операндов. Операнды могут быть двумя регистрами общего назначения или регистром общего назначения и ячейкой памяти. Если используется операнд в памяти, то на время выполнения команды автоматически выставляется сигнал блокировки шины LOCK#, причем это делается независимо от наличия префикса блокировки и текущего значениия IOPL.

Инструкция CMPXCHG (Сравнить и заменить)

Сравнивает значение регистров AL, AX, EAX или RAX c первым операндом (операнд назначения). Если два значение равны, второй операнд (операнд источник) загружается в операнд назначнения. В противном случае операнд назначение загружается в регистры AL, AX, EAX или RAX. RAX регистр доступен только в 64-битном режиме.

Эта инструкция может быть использована с префиксом LOCK чтобы позволить быть выполненной атомарно. Для упрощения интерфейса с шиной процессора операнд-назначения принимает цикл записи, не обращая внимания на результат сравнения. Операнд-назначения записывается обратно, если сравнение неудачно, и операнд-источник записывается в операнд-назначение в противном случае. (Процессор никогда не производит блокировки чтения без блокировки записи.)

Порядок

Чтение может быть переупорядочено с ранее произведенной записью для различных ячеек памяти

Процессор 0 Процессор 1 
mov [_x], 1
mov r1, [_y]
 
mov [_y], 1
mov r2, [_x]
Начальные значения: x = y = 0
r1 = 0 и r2 = 0 разрешено

Однако переупорядочение чтения с предидущей записью для одной ячейки памяти - запрещено.

Процессор 0 
mov [_x], 1
mov r1, [_x]
Начальные значения: x = 0
r1 = 0 запрещено

Предтвращение переупорядочения компилятором 

asm volatile("" ::: "memory");

Предотвращение переупорядочения CPU 

asm volatile("mfence" ::: "memory");

Пример 

/*
gcc -std=gnu11 -O2 -lpthread reordering.c -o reordering 
*/
#include <stdio.h>
#include <pthread.h>

pthread_barrier_t barrier;

volatile int X, Y;
int r1, r2;

void* threadA_function(void* arg)
{
    pthread_barrier_wait(&barrier);
    X = 1;
    asm volatile("" ::: "memory"); // prevent compiler reordering
    r1 = Y;
    return NULL;
}

void* threadB_function(void* arg)
{
    pthread_barrier_wait(&barrier);
    Y = 1;
    asm volatile("" ::: "memory"); // prevent compiler reordering
    r2 = X;
    return NULL;
}

int main()
{
    pthread_t thrA, thrB;

    for (int count=0; ; count++) {
        
        // initial state
        X = Y = 0;
        pthread_barrier_init(&barrier, NULL, 2);

        pthread_create(&thrA, NULL, threadA_function, NULL);
        pthread_create(&thrB, NULL, threadB_function, NULL);

        pthread_join(thrA, NULL);
        pthread_join(thrB, NULL);

        if( (r1 == 0) && (r2 == 0) ){
            printf("%7u> r1=%d, r2=%d (X=%d, Y=%d)\n", count, r1, r2, X, Y);
        } 
   }
   return 0;
}

Видимость

Процессор 0 Процессор 1 
mov [_x], 1
mov r1, [_x]
mov r2, [_y]
 
mov [_y], 1
mov r3, [_y]
mov r4, [_x]
Начальные значения: x = y = 0
r2 = 0 и r4 = 0 разрешено

Пример 

/*
gcc -std=gnu11 -O2 -lpthread visibility.c -o visibility 
*/
#include <stdio.h>
#include <pthread.h>

pthread_barrier_t barrier;

volatile int X, Y;
int r1, r2, r3, r4;

void* threadA_function(void* arg)
{
    pthread_barrier_wait(&barrier);
    X = 1;
    r1 = X;
    r2 = Y;
    return NULL;
}

void* threadB_function(void* arg)
{
    pthread_barrier_wait(&barrier);
    Y = 1;
    r3 = Y;
    r4 = X;
    return NULL;
}

int main()
{
    pthread_t thrA, thrB;

    for (int count=0; ; count++) {
        
        // initial state
        X = Y = 0;
        pthread_barrier_init(&barrier, NULL, 2);

        pthread_create(&thrA, NULL, threadA_function, NULL);
        pthread_create(&thrB, NULL, threadB_function, NULL);

        pthread_join(thrA, NULL);
        pthread_join(thrB, NULL);

        if((r2 == 0) && (r4 == 0)){
            printf("%7u> r2=%d, r4=%d (X=%d, Y=%d)\n", count, r2, r4, X, Y );
        }
   }
   return 0;
}
